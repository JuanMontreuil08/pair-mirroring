## Problem Statement

Remote workers in LATAM use Wallbit as their neobank and investment account. They
already have WhatsApp/Telegram groups where they talk informally about money. But
when it comes to investing together, they have no tool built for them:
- Investment clubs (bivio, myICLUB) are 2005-era accounting tools with no mobile UX
- Social investing apps (Blossom, getquin) don't support Wallbit
- Hedge.io is US-only, requires a new brokerage account, forces equal amounts, has no AI
- Commonstock ($34M raised) was acquired by Yahoo and shut down

The FBI reported a 300% increase in investment club scams in 2025 — not because
people don't want peer investing, but because the legitimate product doesn't exist
for this demographic.

## Target User

LATAM remote workers who:
- Already use Wallbit for their US investments
- Already have a Telegram group with friends/colleagues where they talk about money
- Want to invest together but don't want to pool money or open a new account
- Currently coordinate via informal chat with no structure and no data

---

## Premises

1. No pooled capital = no regulatory burden. Same legal model as eToro CopyTrading —
   each member owns their account, the platform reads and synthesizes. Not a fund.

2. Telegram is the right channel for the hackathon demo. WhatsApp has higher LATAM
   penetration but requires business verification (days). Telegram Bot API is instant.
   For production: swap to WhatsApp. For demo: Telegram is identical UX.

3. Wallbit read-only API is sufficient for the intelligence layer. Trade execution
   is simulated in the demo ("Trade ejecutado — simulado"). The value is the AI
   reasoning before the trade, not the execution itself.

4. Private negotiation per member is better than group negotiation. Each member's
   OpenClaw agent deliberates privately in DM. Group chat stays clean — only the
   final agreed outcome surfaces there. No social pressure, no privacy leaks.

5. Axes 3+4+5 are the only ones that need to be demo-able. Axes 1, 2, 6, 7 can
   be described in the pitch but don't need working implementations on demo day.

## Architecture

### Command set

| Command | Where | Does what |
|---------|-------|-----------|
| `/pair-mirroring` | Group | Session dashboard — pod status, combined portfolio, active proposals, insights |
| `/propose TICKER AMOUNT` | Group | Starts a trade negotiation across all members |
| `/status` | Group | Shows pending proposals only |
| `/leave` | Group | Member disconnects from pod |
| `/start` | DM only | Onboarding — generates magic link to connect Wallbit account |

### `/pair-mirroring` — Session Dashboard

The primary entry point for the group. Members type this whenever they want a snapshot
of what's going on. It does NOT gate other commands — `/propose` always works
independently.

**Response format:**
```
🪞 Pair Mirroring — Los Remotos

👥 Miembros (3/3 conectados)
  • Marcos (BA) ✅
  • María (CDMX) ✅
  • Juan (SP) ✅

💼 Portfolio combinado: $14,320
  Tech 73% | Cash 12% | International 8% | Bonds 7%

📊 Insights del pod
  🚨 Alta concentración en tech (73%) — sin exposure a bonos ni emergentes
  💡 Outlier: María +4.3% esta semana vs pod +2.1%
  💤 $1,720 cash idle combinado sin alocar

📋 Propuestas activas
  • NVDA $300 — negociando (2/3 respondieron) [Ver →]

Usa /propose TICKER MONTO para proponer un trade.
```

**Data sources for this command:**
- Pod name + member list → Supabase `pods` + `pod_members`
- Portfolio totals → Wallbit `GET /balance/stocks` per member (parallel)
- Cash → Wallbit `GET /balance/checking` per member (parallel)
- Active proposals → Supabase `proposals` WHERE status IN ('pending','negotiating')
- Insights → derived from combined portfolio (sector %, outlier detection, cash idle)

**Insight logic (rule-based, no AI needed for this command):**
- Tech concentration alert: if combined tech exposure > 60%
- Cash idle alert: if combined cash > 15% of total portfolio
- Outlier detection: member whose weekly return deviates >2% from pod average
- Bond/international gap: if either allocation = 0%

**Performance note:** fetches Wallbit data for all members in parallel (`Promise.all`).
For 3 members this takes ~1-2s. Add "🔄 Actualizando..." as immediate reply,
then edit with the full response once data is ready (Telegram supports message editing).

### Core concept (MVP scope: Axes 3, 4, 5)

**Eje 3 — Telegram-native:**
A Telegram bot is added to an existing group. Commands in the group trigger
group-level actions. Each member also has a private DM with the bot for sensitive
data (API key connection, personal analysis, negotiation).

**Eje 4 — Individual Wallbit context:**
Each member connects their Wallbit account via a DM magic link. The bot reads
their full portfolio (balance, positions, transactions) — not just the pod slice.
When a trade is proposed, each member receives a personalized DM with impact
analysis specific to their actual holdings.

**Eje 5 — Private negotiation, public outcome:**
Negotiation does NOT happen in the group chat. Each member's OpenClaw agent
deliberates with them privately in DM. When all members have a decision, the
orchestrator synthesizes and posts the final outcome to the group.

### Negotiation flow (Eje 5 — the "whoa" feature)

```
Group: "/propose NVDA 300"
          ↓
Bot in group: "Propuesta recibida: NVDA $300 total.
               Split proporcional: Marcos $150 / María $90 / Juan $60.
               Cada uno recibe análisis privado ahora..."
          ↓
OpenClaw spawns 3 parallel agents (one per member)

  Agent A → Marcos DM:
    "NVDA hoy. Ya tenés 18% en tech — esto te lleva a 26%.
     ¿Aprobás? ✅ ❌ ⚙️ (proponer alternativa)"
    Marcos: ✅

  Agent B → María DM:
    "NVDA hoy. Tu cash baja a $140 — mencionaste reservar $500 para marzo.
     P/E actual: 67x vs histórico 45x.
     ¿Aprobás? ✅ ❌ ⚙️"
    María: ❌
    Agent: "¿Valuación o timing?"
    María: "Concentración en tech"
    Agent: "Si es tech-concentración, ¿QQQ en vez de NVDA solo?
            Spreads el riesgo en todo el sector. ¿Lo propongo al pod?"
    María: ✅ (counteroffer: QQQ)

  Agent C → Juan DM:
    "NVDA hoy. Tu tech exposure es solo 4% — trade hace sentido para tu perfil.
     ¿Aprobás? ✅ ❌ ⚙️"
    Juan: ✅

          ↓
OpenClaw orchestrator: 2 aprueban NVDA, 1 propone QQQ
→ Segunda ronda: envía contrapropuesta QQQ a Marcos y Juan

  Agent A → Marcos DM: "María prefiere QQQ por diversificación. ¿Cambiás?"
  Marcos: ✅

  Agent C → Juan DM: "María prefiere QQQ. ¿Aceptás?"
  Juan: ✅

          ↓
Todos de acuerdo: QQQ, splits proporcionales
          ↓
Group: "✅ Acuerdo del pod: QQQ
        Marcos: $150 | María: $90 | Juan: $60
        [Ejecutar (simulado)] [Cancelar]"
```

**Timeout rule:** Si un miembro no responde en 24h, el agente vota según su
comportamiento histórico ("Juan suele aprobar trades <$100 — auto-aprueba").
El miembro recibe notificación de la decisión tomada en su nombre.

### Technical architecture

```
Telegram Bot (Telegraf / Node.js)
    ↓
Group commands: /propose, /status, /leave
DM: magic link onboarding, personal analysis, negotiation thread
    ↓
OpenClaw orchestrator
    ↓
Per-member agents (parallel):
  - Context: full Wallbit portfolio (balance + positions + transactions)
  - Prompt: personalized analysis + negotiation for this specific member
  - Output: { decision: "approve" | "reject" | "counteroffer", terms: {...} }
    ↓
Synthesis agent:
  - Input: array of per-member decisions
  - Output: consensus | new negotiation round | deadlock
    ↓
Wallbit read API:
  GET /api/public/v1/balance/stocks     → positions
  GET /api/public/v1/balance/checking   → cash
  GET /api/public/v1/transactions       → history (behavioral inference)
  GET /api/public/v1/assets/{symbol}    → sector classification
    ↓
Simulated trade confirmation (no POST /trades — read-only API)
```

**OpenClaw prerequisite:** Confirm Telegram webhook + OpenClaw integration on Day 1.
Fallback if OpenClaw setup > 4h: sequential Claude API calls. UX is identical.

**Per-member agent output contract:**
```json
{
  "member_id": "string",
  "decision": "approve" | "reject" | "counteroffer",
  "counteroffer": { "symbol": "string", "amount": number } | null,
  "reasoning": "string",
  "risk_flags": ["string"]
}
```

## Build Order (4 days, axes 3+4+5 only)

Assumes 1-2 engineers with existing Mirror codebase familiarity.

**Day 1 — Foundation**
- Telegram bot via Telegraf (bot live in test group in <1h)
- Wallbit read API: pull balance + portfolio + transactions for one account
- Pod data model: pod_id, members[] with {telegram_user_id, wallbit_api_key}
- /start in DM → magic link → member connects Wallbit

**Day 2 — Intelligence layer (Eje 4)**
- Per-member Claude agent: full Wallbit portfolio as context
- OpenClaw orchestration (or sequential fallback)
- /propose TICKER AMOUNT in group → proportional splits calculated
- Personalized DM per member with impact analysis

**Day 3 — Negotiation (Eje 5)**
- Inline keyboard: ✅ ❌ ⚙️ in each member's DM
- ❌ triggers: agent asks why → pulls market data → proposes counteroffer
- Orchestrator tracks state across negotiation rounds
- Consensus → group final message + simulated execution

**Day 4 — Demo rehearsal**
- 3 Telegram accounts + 3 Wallbit accounts (or mock data)
- Timeout/auto-vote logic for unresponsive members
- Split-screen demo prep (group view + one member's DM side by side)
- Fix friction points, rehearse 3-minute flow
